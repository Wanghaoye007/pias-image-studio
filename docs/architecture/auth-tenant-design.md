# PIAS 身份与租户授权设计

## 目标

为现有单机工作台建立可替换的服务端身份边界，覆盖密码登录、强制 MFA、会话生命周期、角色权限和 Tenant/Project 隔离。该边界后续可替换为 OIDC，但业务 API 只依赖统一的 `AuthContext`，不直接信任前端角色或操作者字符串。

## 方案选择

1. 外部 OIDC：生产成熟，但当前缺少企业 IdP、回调域名和客户端凭证，无法独立完成本地验收。
2. 内置身份服务：使用 Node 标准加密能力，支持 scrypt、TOTP、随机会话和 RBAC；本轮采用，后续通过版本化适配器替换。
3. 反向代理身份头：实现快，但无法可靠覆盖 MFA、会话撤销和直接 API 越权测试，不采用。

## 核心模型

- `AuthUser`：用户、Tenant、角色、项目范围、成员状态、密码哈希、MFA Secret 和 MFA 状态。
- `AuthSession`：随机 token 的 SHA-256 哈希、用户、创建时间、最近活动时间、MFA 验证时间；服务端不保存明文 token。
- `AuthContext`：通过会话生成的可信用户、Tenant、角色和项目范围。
- `Permission`：项目查看/编辑、素材编辑、任务创建/取消、审核、生产导出、用量查看、审计和成员管理。
- `StudioState` 命令守卫：对整包状态写入执行差异检查，审核流、任务、素材、场景、导出分别映射到对应 Permission；新增审计 actor 由会话用户 ID 覆盖，既有审计历史不可删除或篡改。
- Fal 任务所有者：任务提交时保存会话用户 ID；Creator 只能取消自己创建的任务，Owner/Admin 可使用 `job.cancel_any`。

## 安全规则

- 密码不少于 12 位，并拒绝常见密码；持久化仅允许 scrypt 哈希格式。
- 连续 5 次密码失败后渐进锁定，最长 15 分钟；成功登录后清零。
- Owner、Admin、Platform Operator 必须完成 TOTP MFA 才能取得业务会话。
- 会话闲置 30 分钟失效，绝对时长 12 小时；成员停用后下一次校验立即失效。
- 会话 token 使用 256 bit 随机值，服务端仅保存哈希，并支持单会话和全会话撤销。
- Tenant 不一致统一判定为资源不可见；私有项目还必须满足项目成员范围。Owner/Admin 可查看本 Tenant 全部项目，Platform Operator 默认不能查看客户内容。
- 权限判断只接受服务端 `AuthContext`，不得接受客户端传入的角色、Tenant 或 actor。
- 客户端可携带展示用 actor，但服务端保存前会覆盖新增事件、审核人和采用人；客户端旧 actor 与已存记录不一致时，以服务端历史为准，不会反向覆盖可信记录。
- 未知状态命令、非法审核跳转、缺少匹配审计事件、删除或修改既有审计记录均拒绝写入。

## 接入顺序

1. 已完成：身份、MFA、会话和授权核心及单元测试。
2. 已完成：`/api/auth/*` 会话接口、HttpOnly Cookie 与 CSRF 防护。
3. 已完成：StudioState、素材和 Fal API 已进入认证中间件，三类持久化均按 Tenant/Project 路由存储并具备迁移回滚。
4. 已完成：登录/MFA/退出 UI、路由级素材/Fal 权限与状态命令级 RBAC 已接入。
5. 已完成：双 Tenant IDOR、角色矩阵、停用成员/旧会话失效，以及 Creator 提交/越权审批/Reviewer 审批 E2E。

## 回滚

安全核心为新增模块，未接入 API 前不改变现有运行路径。接入阶段按中间件开关逐 API 迁移；若发现兼容问题，可回退到上一构建产物和原状态文件，身份配置与业务状态分开存储，不修改现有业务快照格式。
