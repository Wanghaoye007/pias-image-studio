# AI 生成业务工具验收与自查标准

## 1. 使用方式

本标准用于 AI 辅助开发业务系统的项目验收、版本复验和缺陷自查。验收必须以业务规则、完整代码、运行证据和数据证据为输入，不能仅依据界面演示或开发者口头说明。

执行顺序固定为：需求对齐 -> P0 底线检查 -> 四阶自查 -> 缺陷复现 -> 分级判定 -> 整改复验。

## 2. 自查前置条件

以下任一项缺失时，自查结论只能标记为“证据不足”，不得判定验收通过：

1. 完整业务规则原文，包括状态流转、角色权限、计算口径、逆向操作和异常处理。
2. 对应模块的完整代码，包括字段定义、接口声明、数据迁移和外部依赖。
3. 可运行环境、测试数据、日志入口及关键链路的复现步骤。
4. AI 先用自己的语言复述业务目标、核心对象、主流程、逆向流程和验收边界，并由需求方确认。

## 3. 缺陷等级

| 等级 | 定义 | 典型示例 | 处理要求 |
| --- | --- | --- | --- |
| P0 阻断 | 核心业务不可用，或造成数据丢失、越权、严重错账 | 主链路崩溃；刷新丢失已提交业务数据；未授权用户可审批或查看其他租户数据 | 立即停止验收和上线，修复并人工复现验证 |
| P1 严重 | 主链路可运行，但关键业务规则错误或重要场景无处理 | 非法状态跳转；重复扣费；取消后账务错误；关键逆向流程缺失 | 正式上线前必须清零 |
| P2 一般 | 局部功能受损，有替代路径，不破坏核心数据 | 次要按钮无响应；错误提示不清；筛选条件丢失 | 纳入当前或下一版本修复并评估风险 |
| P3 优化 | 不影响业务正确性的体验或维护性问题 | 文案、对齐、可读性、低风险重复代码 | 排期优化 |

## 4. 四层验收标准

### 4.1 底线可用性（P0 否决项）

以下任一项成立，结论直接为“不予验收”：

- 主干业务流程无法从输入完整走到可用结果。
- 出现白屏、持续接口错误、死循环、卡死或不可恢复的数据丢失。
- 缺少必要的身份认证、权限校验或租户/项目数据隔离。
- 相同输入与确定性参数重复执行时，核心业务字段出现无法解释的错乱。
- 账务、额度、状态或业务主数据在刷新、重试、取消后不一致。

### 4.2 业务逻辑核心验收

逐模块检查以下内容，并为每项提供代码、测试、接口响应或数据库记录作为证据：

- 正向流程：创建、编辑、提交、处理、查看结果、导出等主路径闭环。
- 逆向流程：撤回、驳回、取消、删除、退回修改、重试和恢复。
- 状态规则：允许的状态转换、操作者、前置条件和副作用均明确。
- 分支规则：多条件优先级、空值、超限、非法输入、重复操作均有确定结果。
- 权限规则：每个角色的查看、创建、修改、审批、导出和管理权限由服务端执行。
- 可追溯性：AI 结果记录输入、模型、参数、版本、时间、操作者和上游结果。

### 4.3 系统健壮性

- 输入错误以业务语言提示，不泄露底层堆栈或密钥。
- 重复提交具有幂等保护；并发更新有冲突控制或确定的合并策略。
- 外部依赖失败、超时、限流和网络中断可重试、可取消、可追踪。
- 页面刷新、进程重启后，已提交任务和业务数据可恢复。
- 失败操作不会遗留脏数据、幽灵占用、重复扣费或非法中间状态。

### 4.4 数据与状态一致性

- 前端展示、接口返回、持久化数据和审计记录使用同一业务事实。
- 状态迁移只能通过定义好的命令执行，不允许界面直接篡改最终状态。
- 金额、额度、数量和统计指标具有唯一口径，预占、结算、释放可对账。
- 关联对象同步更新，删除和取消不会产生失联引用。
- 审计记录包含主体、动作、时间、对象、前后状态和追踪标识，且不可被普通用户修改。

## 5. 四阶自查流程

### 第一阶：业务逻辑完整性

1. 按角色、对象、状态、动作和异常五个维度建立场景矩阵。
2. 穷举正向、逆向、跨角色、多条件和重复操作场景。
3. 对照代码与测试，将每个场景标记为“已实现 / 部分实现 / 未实现 / 逻辑错误 / 证据不足”。
4. 先修复业务设计缺失，再进入代码细节检查。

### 第二阶：代码逻辑与显性 Bug

检查类型和语法、变量生命周期、条件优先级、循环终止、异步竞态、接口契约、错误处理、资源释放及敏感信息暴露。每个问题必须给出可执行修复方案和最小回归测试。

### 第三阶：边界与异常场景

专项覆盖空值、零值、极值、超限、非法格式、重复提交、快速连续操作、并发编辑、刷新恢复、依赖超时、部分成功、取消与重试。对每个场景记录触发步骤、期望结果和实际结果。

### 第四阶：数据与状态一致性

绘制状态转换表和数据血缘，核对每次操作的前端、接口、存储、审计和账务结果。重点反证非法跳转、重复结算、状态回退、孤儿数据和跨租户引用。

## 6. 分级判定算法

判定按最严重问题执行，不允许用问题数量抵消严重程度：

| 结论 | 判定条件 | 处理方式 |
| --- | --- | --- |
| 不予验收（红色） | 存在任意 P0；主链路不闭环；生产必需的认证、隔离或持久化缺失 | 驳回整改，禁止生产使用 |
| 可试用不可正式上线（黄色） | 无 P0，但存在 P1；或边界、异常、逆向场景存在系统性缺失 | 仅限受控演示或小规模试用 |
| 验收通过（绿色） | P0/P1 清零；约定范围场景均有证据；自动化检查全部通过；数据和状态闭环 | 可按发布流程上线 |

“绿色”还必须同时满足：所有必选检查项为“已通过”，没有“证据不足”；P0/P1 缺陷经过人工复现确认关闭；测试、构建和数据迁移检查通过。

## 7. 强制输出格式

每次自查必须包含：

1. 需求对齐复述。
2. 验收范围、环境、版本和证据来源。
3. 场景覆盖统计。
4. 红黄绿结论及触发依据。
5. 结构化问题清单。
6. 无问题维度的明确声明。
7. 整改顺序、复验条件和剩余风险。

问题清单字段不得删减：

| 字段 | 说明 |
| --- | --- |
| 问题编号 | 全局唯一，例如 `CS-P0-001` |
| 问题类型 | 业务、代码、边界、数据、状态、权限或可追溯性 |
| 所在位置 | 文件与行号、接口、数据表或页面 |
| 触发条件 | 可复现步骤和输入 |
| 实际结果 | 当前系统行为 |
| 期望结果 | 业务规则要求 |
| 影响等级 | P0/P1/P2/P3 |
| 修复方案 | 可落地的代码、数据或设计修改 |
| 验证方式 | 自动化测试与人工复验步骤 |
| 证据状态 | 已复现、静态证据、推断或证据不足 |

禁止使用“整体正常”“基本没问题”“应该可以”等无证据结论。某个维度未发现问题时，必须明确写出“该维度无问题”，并列出执行过的检查。

## 8. 复验门槛

- P0/P1 必须逐项关联修复提交、回归测试和人工复现记录。
- 缺陷关闭后重新执行完整主链路，不只回归单个故障点。
- 涉及数据或状态的修复必须验证历史数据迁移、并发和幂等。
- 涉及 AI 外部服务的修复必须覆盖成功、失败、超时、取消、重试和部分成功。
- 验收报告、命令输出和关键截图纳入版本交付物。
